# SpatialIndexV2 — Отчёт о проблемах
Дата: 2026-02-21
Тест: 1M entities, 1000 queries, 8 update passes, Profile /O2, 128×128 chunks

## Результаты теста

| Метрика | V1 | V2 | Отношение |
|---------|----|----|-----------|
| Query avg | 5233µs | 897µs | V2 быстрее в **5.8x** |
| Query p99 | 17557µs | 2996µs | V2 быстрее в **5.9x** |
| Update avg | 41ms | 77ms | V2 медленнее в **1.9x** |
| Update p99 | 59ms | 81ms | V2 медленнее в **1.4x** |
| Overflow | 0 | 0 | Чисто |

## Проблема 1 — Update медленнее V1 в 2x

**Статус:** Не проблема для пошаговой стратегии. Для реалтайма — потенциальный риск.

**Факты:**
- V2 update avg=77ms на 8 проходов = **~9.6ms за один полный update** 1M сущностей
- В пошаговой стратегии update происходит один раз при обработке хода, не 60 раз в секунду
- 9.6ms на обновление позиций 1M юнитов — абсолютно приемлемо для Civ/Paradox-масштаба
- Даже если обновлять 10 раз за ход — это 96ms, незаметно для игрока

**Причина медленного update в V2:**
- V2 при updateEntity делает remove из старых чанков + insert в новые
- При пересечении границы чанка — двойная запись
- V1 — flat grid, O(1) remove/insert без чанковой логики

**Пути решения (если понадобится):**
- Dirty-flag батчинг: накапливать изменения, применять пачкой раз в N тиков
  - Плюс: амортизирует стоимость update
  - Минус: позиции в индексе отстают от реальных на N тиков
- Разделить "переместился в новый чанк" и "переместился внутри чанка" — второй случай не требует full update
  - Плюс: значительное ускорение при локальных движениях
  - Минус: усложняет логику updateEntity

**Вывод:** Не фиксировать как баг. Вернуться если реалтайм-события (битвы) потребуют массового update.

## Проблема 2 — Candidates per query avg=15748

**Статус:** Артефакт харнеса, не баг spatial index.

**Факты:**
- Харнес генерирует query AABB случайного размера до 25% ширины мира
- 25% от 128 chunks = 32 chunks в одном запросе → огромная область
- В реальной игре viewport-запрос = 3×3 видимых чанка → кандидатов будет ~200-300

**Что значит высокое число кандидатов:**
- Spatial index возвращает всех кандидатов в области — culling (отсев по точному AABB) делается выше
- Если кандидатов много — culling работает больше, но это нормально при широком запросе
- Для глобальной стратегии с minimap-запросами (весь мир) это реальная нагрузка

**Реальная проблема culling:**
До V2 culling в RenderSystem был агрессивным (viewport frustum). После введения SpatialStreamingSystem и V2 culling-pipeline изменился. Нужно проверить что `maxVisibleSprites` и `budgets(visible=576)` в конфиге корректно ограничивают финальный список для рендера.

**Пути решения:**
- Добавить в харнес реалистичный размер query (viewport 3×3 chunks) как отдельный сценарий
  - Плюс: репрезентативные цифры для игрового случая
  - Минус: придётся менять харнес
- Добавить two-phase culling: spatial query → грубый AABB → точный frustum
  - Плюс: снижает нагрузку на дорогие операции в hot path
  - Минус: ещё один слой логики

## Проблема 3 — Харнес не тестирует реальный игровой паттерн

**Статус:** Ограничение инструмента, не баг движка.

**Факты:**
- Харнес: синтетические случайные AABB, однородное распределение по всему миру
- Реальная игра: кластеры (города, армии), горячие зоны, большинство entities неподвижны между ходами
- Харнес не тестирует стриминг (load/unload чанков при движении камеры)
- Харнес не тестирует deterministic update order (критично для мультиплеера)

**Пути решения:**
- Добавить в харнес сценарий "кластерное распределение" (Poisson disk sampling)
- Добавить сценарий "движение камеры" с вызовами setWindowOrigin
- Добавить сценарий "95% entities неподвижны, 5% двигаются" — реальный Civ-паттерн
- Всё это — Phase 2 (Google Tests) или расширение харнеса отдельным таском

---

## Проблема 4 — V1 остаётся в кодовой базе как dead code

**Статус:** Технический долг, не баг.

**Факты:**
- V1 используется только как oracle в харнесе (correctness baseline)
- В игровом коде V1 не используется нигде
- Удалить V1 нельзя пока нет покрытия V2 unit-тестами (Phase 2)

**План:**
- Phase 2 (Google Tests): написать unit-тесты для V2 покрывающие граничные случаи
- После Phase 2: удалить V1 из production кода, оставить только в харнесе или удалить полностью
- После удаления V1: харнес теряет correctness-сравнение, остаются только timing-сценарии

## Итог

V2 **прошёл тест успешно**:
- 5.8x быстрее V1 по query на 1M сущностей
- Overflow нулевой при любом масштабе
- Все сценарии корректности passed
- Приемлем для пошаговой стратегии Civ/Paradox-масштаба

Критических проблем нет. Все пункты выше — улучшения и технический долг для Phase 2+.
