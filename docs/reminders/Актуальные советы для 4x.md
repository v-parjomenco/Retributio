## 1.1 🔴 CMake (СЕЙЧАС - у вас Linux)

### КОГДА
**ПРЯМО СЕЙЧАС** - вы установили Linux и хотите пробовать на нём.

### Проблема
VS .sln/.vcxproj только для Windows. Для Linux нужен CMake.

### Решение
Добавить CMakeLists.txt параллельно с VS-проектом:
- VS-проект продолжает работать (основная разработка)
- CMake для Linux/CI постепенно стабилизируется
- Режим "параллельно" до полной стабильности CMake

### Польза для 4X
- CI для performance regression
- Linux server builds для dedicated 4X server
- Единообразная сборка при появлении второго проекта

## 1.3 🔴 Tick-based simulation (СЕЙЧАС - детерминизм)

### КОГДА
**УЖЕ В SKYGUARD** - детерминизм полезен для тестов и replay даже в мини-игре.

### Проблема
TimeService завязан на wall-clock → недетерминизм.

### Решение (минимальная версия для SkyGuard)
```cpp
class TimeService {
    std::uint64_t mTickNumber = 0;
    float mFixedDt = 1.f / 60.f;
    
    void tick() {
        ++mTickNumber;
        // Симуляция на фиксированном шаге
    }
    
    float getSimTime() const {
        return mTickNumber * mFixedDt;
    }
};
```

### Применение в SkyGuard
- Replay controls для debug (pause/step/rewind)
- Детерминистичные тесты
- Базовая инфраструктура для 4X

### Польза для 4X
- **КРИТИЧНО**: детерминизм/replay/multiplayer
- Фундамент для turn-based системы
- Debug AI: воспроизведение партий

## 1.5 🟡 Multi-frequency update (базовая версия СЕЙЧАС)

### КОГДА
**УЖЕ В SKYGUARD** - tactical systems vs debug overlay разные частоты.

### Решение (простая версия)
```cpp
class ISystem {
    virtual void update(World&, float dt) = 0;
    virtual std::uint32_t updateFrequency() const { return 1; } // каждый тик
};

class DebugOverlaySystem : public ISystem {
    std::uint32_t updateFrequency() const override { 
        return 4; // обновление раз в 4 тика (~15 FPS)
    }
};

// В SystemManager:
void update(World& world, float dt) {
    for (auto& sys : mSystems) {
        if ((mTickNumber % sys->updateFrequency()) == 0) {
            sys->update(world, dt);
        }
    }
    ++mTickNumber;
}
```

### Применение в SkyGuard
- DebugOverlay: 15 FPS (экономия sprintf/draw)
- ScalingSystem: только при resize event
- Практика архитектуры

### Польза для 4X
- **КРИТИЧНО**: AI systems на разных частотах
- Tactical: 60 FPS, Strategic AI: 2 FPS, Economy: 0.1 FPS

## 1.6 🟢 Profiling infrastructure (СЕЙЧАС - измерять ДО оптимизаций)

### КОГДА
**УЖЕ В SKYGUARD** - нужно знать, где узкие места.

### Решение
```cpp
// Расширение текущего SFML1_PROFILE
#if defined(SFML1_PROFILE)
    struct ScopedTimer {
        const char* name;
        std::chrono::steady_clock::time_point start;
        
        ~ScopedTimer() {
            auto end = std::chrono::steady_clock::now();
            auto us = std::chrono::duration_cast<std::chrono::microseconds>(end - start);
            ProfileLog::record(name, us.count());
        }
    };
    
    #define PROFILE_SCOPE(name) ScopedTimer _timer{name}
#else
    #define PROFILE_SCOPE(name) do {} while(0)
#endif
```

### Применение в SkyGuard
- MovementSystem: сколько µs на 10k entities
- RenderSystem: gather/sort/draw breakdown
- SpatialIndex: query cost

### Польза для 4X
- **КРИТИЧНО**: знать, где оптимизировать
- Regression tracking при масштабировании
- CI integration

## 2.1 🔴 Resource System без singleton

### КОГДА
При появлении **ПЕРВОГО** из:
- Второй game-проект в репо (не SkyGuard)
- Хот-релоад ресурсов/конфигов
- Моды с overlay-реестрами
- Юнит-тесты с несколькими контекстами
- Редактор рядом с игрой

### Проблема
Meyer singleton ResourceRegistry + static caches → блокер для:
- Несколько контекстов ресурсов в одном процессе
- Hot-reload без сайд-эффектов
- Моды

### Решение
```cpp
// Было:
// static ResourceRegistry& instance();
// static ResourcePaths& instance();

// Станет:
class ResourceContext {
    ResourceRegistry registry;
    ResourceManager manager;
public:
    ResourceContext(std::span<const ResourceSource> sources);
};

// В Game:
std::unique_ptr<ResourceContext> mResources;
```

### Польза для 4X
- Моды: overlay resource contexts
- Тесты: параллельные контексты
- Редактор: независимый от игры

## 2.2 🟡 Save/Load система (EnTT Snapshot)

### КОГДА
При старте реального 4X проекта (нужны сохранения).

### Проблема
Нет сериализации game state.

### Решение
```cpp
// EnTT native snapshot (binary, быстро, детерминировано)
entt::snapshot snapshot{world.registry()};

// Output archive
std::ofstream output{"save.bin", std::ios::binary};
cereal::BinaryOutputArchive archive{output};

// Сериализация всех компонентов
snapshot.entities(archive)
    .component<TransformComponent, 
               VelocityComponent,
               ProvinceStateComponent,
               // ...
              >(archive);
```

### Польза для 4X
- **КРИТИЧНО**: save/load 500k entities
- Быстрее JSON парсинга
- Детерминировано
- Версионирование (migration между версиями)

## 3.1 🔴 Turn System + Async AI

### КОГДА
При реализации основного 4X gameplay loop.

### Решение
```cpp
void processTurn() {
    // 1. Player input (уже применён)
    
    // 2. AI turns (параллельно, 5-10 сек бюджет)

    mDiplomacySystem.tick(mWorld);

    // 5. Cleanup
    mWorld.destroyEntities(mPendingDestroys);
}
```

### Польза для 4X
- **КРИТИЧНО**: обработка 2000 AI игроков
- Предсказуемое время хода

## 3.2 🟢 Offline JSON→binary compilation

### КОГДА
При добавлении больших датасетов (провинции, события, history).

### Решение
```
JSON (human-readable configs)
  ↓ offline tool при сборке
Binary blob (provinces.bin, events.bin)
  ↓ runtime
Прямая десериализация (memcpy в структуры)
```

### Преимущества
- Загрузка 10k провинций: миллисекунды вместо секунд
- Валидация при сборке (ошибки не в рантайме)
- Детерминированное представление
- EnTT Snapshot совместимость

### Польза для 4X
- Быстрый старт партии
- Моды: компиляция при установке
- CI: автоматическая валидация данных

## 3.3 🟡 Archetype Storage (ТОЛЬКО по профайлингу)

### КОГДА
**ТОЛЬКО ЕСЛИ** профайлинг показывает cache misses в AI iteration.

### ⚠️ ВНИМАНИЕ
EnTT уже оптимизирован для cache locality. Archetype может быть **НЕ НУЖЕН**.

### Как проверить
```cpp
// Профайлер должен показать:
// - L1 cache misses > 20% в AI loop
// - Memory bandwidth saturation

PROFILE_SCOPE("AI_Province_Iteration");
for (auto [entity, province, state] : view.each()) {
    // Если здесь cache misses - рассмотреть archetype
}
```

### Решение (если действительно нужно)
```cpp
// EnTT groups (почти как archetype, но легковеснее)
auto group = registry.group<ProvinceState>(entt::get<ProvinceData>);

// Или custom archetype storage (много работы)
```

### Польза для 4X
- Потенциально: 2-3x speedup AI iteration
- НО: сложность + может быть ненужно

## 4.2 🟢 Advanced multi-frequency (AI time-slicing)

### КОГДА
Когда 2000 AI игроков не укладываются в бюджет хода (5-10 сек).

### Решение
```cpp
class AIScheduler {
    // Распределение 2000 игроков на несколько тиков
    void processAllAI(World& world) {
        const int aiPerTick = 200;
        for (int tick = 0; tick < 10; ++tick) {
            processAIBatch(world, tick * aiPerTick, aiPerTick);
            // UI остаётся responsive
        }
    }
};
```

### Польза для 4X
- UI responsive во время AI хода
- Равномерная нагрузка CPU
- Predictable frame time

# КЛЮЧЕВОЙ ПРИНЦИП

> **Делать в SkyGuard всё, что:**  
> 1. Не требует 4X контента (провинции, AI)  
> 2. Полезно и для мини-игры (детерминизм, профайлинг)  
> 3. Фундамент для 4X (tick system, tags, multi-frequency)

> **НЕ делать в SkyGuard:**  
> 1. 4X-специфичное (turn system, provinces)  
> 2. Преждевременные оптимизации (archetype без профайла)  
> 3. "На всякий случай" (X-macro при 2 ресурсах)

**Правило:** "Можно реализовать БЕЗ 4X контента?" → делай в SkyGuard  
**Иначе:** жди реального use-case
