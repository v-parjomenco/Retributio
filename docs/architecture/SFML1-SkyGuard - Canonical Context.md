SFML1 / SkyGuard — Canonical Context (Part 1): SFML1 / SkyGuard is a high-performance 2D engine for gigantic-scale turn-based grand strategy (flagship 4X), flexible enough for side projects (platformers, small RPGs) for rapid development/funding, but optimized and architected specifically for 4X targeting “indie-AAA” quality. Tech stack: C++20, SFML 3.0.2, nlohmann-json 3.12, EnTT 3.16.0. SkyGuard is the first small game/prototype used as a lab to validate architecture, performance, ECS, resources, and UI. Long-term flagship: a very large-scale 2D turn-based 4X / grand strategy with layered simulation, deterministic saves/replay, event system, and data-driven AI.
SFML1 / SkyGuard — Canonical Context (Part 2): Core Architecture Principles: Strict core/game separation — core/ is engine-only, game-agnostic, reusable; game/* (e.g., game::skyguard) is game-specific and never pollutes core. Modularity — each module/file has one clear responsibility; split “god files”; build from small autonomous bricks with minimal dependencies (no mega-modules/spaghetti). Zero-overhead mindset — avoid virtual/type-erasure in hot paths; prefer data-oriented layouts and inlining; when in doubt, keep hot-path cost minimal and predictable (no defensive per-frame/per-entity branching); handle failures at boundaries via logging, fallbacks, or fail-fast where appropriate. Always validate on write, trust on read — validate only at mutation boundaries (loading/authoring/applying config/state transitions); Release must not pay recurring validation costs in hot paths; Debug-only asserts allowed. Data Contract: Code IDs vs JSON Truth — resource identity is defined by the Resource Registry v1 contract: Canonical Key String (namespaced, human-readable, mod-friendly) + StableKey64 (persistence/cross-version stability) + RuntimeKey32 (hot path handle). Critical rule: hot path uses only RuntimeKey32 (no strings/hashes/maps). Compile-time enums are not a core resource contract; allowed only as local convenience in game/prototype code. JSON is the source of truth for content/parameters/mapping (paths, numeric values, tuning/balance, registries, etc.).
SFML1 / SkyGuard — Canonical Context (Part 3): Repository Structure (logical): Single repo containing engine core (include/core, src/core), game layer(s) (include/game/..., src/game/...), and assets/configs split by core/game (assets/core/..., assets/game/...). Also includes tools/scripts and architecture docs. Dev-only utilities may exist inside the game layer (e.g., stress scenes) but never in core; they must not contaminate Release builds and are allowed only in Debug/Profile with appropriate compile-time and/or runtime gating.
SFML1 / SkyGuard — Canonical Context (Part 4): Code Quality & Style hard rules: AAA, commit-ready, production-grade code only; no toy/demo implementations. Indentation: 4 spaces, no tabs. Braces on same line (if (...) {) and always use braces even for single-line statements. Pointer/reference style: T* ptr, T& ref. Target line length ~100 chars (avoid routinely exceeding). .cpp #include order: (1) corresponding header, (2) standard library, (3) SFML headers, (4) other project headers; separate blocks with a blank line; sort alphabetically within each block. Naming: clear, descriptive identifiers (short only for standard cases like dt). Comments & docs: each new/modified header has a single top-of-file banner in English (file path, purpose, used by, related headers, notes); .cpp usually must not have top-of-file banners; no additional banner headers inside files. In-code comments are in Russian, explain intent/invariants/non-obvious decisions; comments may be rewritten but must not be silently removed.
SFML1 / SkyGuard — Canonical Context (Part 5): Resource Layer Rules: Resource identity follows Resource Registry v1: Canonical Key String (canonical ID, namespaced, human-readable, mod-friendly) + StableKey64 (persistence/cross-version stability) + RuntimeKey32 (hot path handle). Canonical Key String format (frozen): namespace.category.name; minimum 3 segments separated by '.'; each segment: [a-z][a-z0-9_-]*; entire key lowercase; allowed chars: [a-z0-9._-]; regex: ^[a-z][a-z0-9_-]*(\.[a-z][a-z0-9_-]*){2,}$; validation is mandatory at registry build (authoring error on failure). Pipeline: resources.json defines registries and load parameters; runtime registry is built at load time with validate-on-write; runtime code trusts resolved handles. Ownership: ResourceManager is the façade (owned by composition root, no globals); ResourceHolder is the cache; registry internals (key storage, indices, path mapping / ResourcePaths-like mapping) are private to the resource subsystem (no external dependency). Render hot path rule: resolve TextureKey (RuntimeKey32) -> sf::Texture via an index-addressed per-frame cache (cache unique textures once per frame); render hot loops must use O(1) handle-based access only (no strings, no hashing, no maps). Per-frame uniqueness caching MUST NOT use associative containers in render hot loops (e.g., unordered_map); use an index-addressed epoch/stamp mechanism keyed by RuntimeKey32.index().
SFML1 / SkyGuard — Canonical Context (Part 6): Logging Policy (severity + categories): Release logs are Info+ only (Debug/Trace off by default, zero perf impact on simulation). Trace is per-frame/tight-loop, very noisy; never Info/Warning. Debug is rare, rate-limited diagnostics; never tight loops. Info is rare milestones safe for Release. Warning is recoverable, actionable, non-spam. Error is malfunction but process may live. Critical is fatal, usually panic/exit. Stable categories: Engine, Config, Resources, ECS, UI, Gameplay, Debug/Performance.
SFML1 / SkyGuard — Canonical Context (Part 7): Directory ↔ Category ↔ Logic: Each directory/module owns one primary responsibility (“one logic”) and has a default logging category aligned with that responsibility. Cross-category logs are allowed only when the meaning genuinely changes; the directory’s default category remains the baseline. Canonical default mapping: core/log/* → Engine; core/compiler/* → Engine; core/config/* → Config; core/utils/json/* → Config; core/resources/* → Resources; core/ui/* → UI; core/ecs/* → ECS; core/time/* → Engine (service init/ownership) or Debug/Performance (profiling/timers/metrics); core/utils/file_loader.* → Engine by default (low-level I/O), but Config if explicitly about config loading outcomes; core/debug/* → Debug/Performance; game/skyguard/* (config) → Config for JSON/config errors, Gameplay for game-meaning outcomes; game/skyguard/* (ecs/game) → Gameplay.
SFML1 / SkyGuard — Canonical Context (Part 8): JSON / Config Loading Contract: JSON parsing+validation helpers: core::utils::json::parseAndValidateCritical / parseAndValidateNonCritical (in json_document.h). Critical mode: LOG_PANIC + terminate on error. Non-critical mode: returns std::nullopt; detailed parsing/validation logging is DEBUG only. Non-critical loaders must emit exactly one WARN when dataOpt is nullopt to describe the semantic outcome (“config not applied, using defaults”); avoid duplicate WARN across layers. Any noexcept parsing helper must not call throwing APIs (e.g., nlohmann::json::get<T>() when it may throw); use type-checked access + finite/range checks (no-throw path) to avoid std::terminate. Three-tier loader policy: Type A (critical, e.g., resource registries): parseAndValidateCritical for any I/O or JSON failure; loader does not add extra ERROR/PANIC logs. Type B (important with safe defaults): I/O failure -> WARN “using defaults” and return defaults; JSON failure -> critical or non-critical depending on whether it’s tuning-only. Type C (dev/cosmetic): I/O failure -> WARN + defaults; JSON failure -> parseAndValidateNonCritical fallback; never crash.
SFML1 / SkyGuard — Canonical Context (Part 9): ECS (EnTT) Core Rules. 1) World / Registry Contract: Backend EnTT (entt::entity, entt::registry). core::ecs::Entity = entt::entity; NullEntity = entt::null. core::ecs::World owns exactly one entt::registry and a SystemManager. No custom Registry, no custom ComponentStorage<T>; all components live in EnTT storage. World API includes: create/destroy/isAlive; add/get/remove component; view/group; addSystem/update/render; registry() is escape hatch only. Wrappers must inline to raw EnTT calls (no virtual/type-erasure in hot paths). Backend strategy: World must not leak EnTT types outside core/ecs (no EnTT in public game-facing APIs). ECS backend may be swapped if profiling proves storage/iteration bottlenecks at flagship scale. 1.5) Systems: Access Declarations & State: Systems declare component access explicitly (read-only vs read-write) to enable future scheduling/parallelization/conflict detection. Systems are essentially stateless (except configuration) and operate on ECS data; external services are injected explicitly (composition root), not hidden globals. 2) ECS Component Contract (performance-first): Components are POD/SoA-friendly: trivially copyable; no custom ctor/dtor; no virtuals. Components must not own heavy objects/resources (forbidden: sf::Sprite, sf::Texture, sf::RenderWindow, file handles, owning smart pointers, etc.). Small trivially-copyable value types OK if no per-entity overhead (e.g., sf::IntRect). Store only IDs, flags, small numeric/enum types, simple POD structs. Deterministic defaults: value-init (T t{}) and/or in-class initializers; for default objects always Type t{} (never Type t;). Sparse/heavy state policy: rare/heavy state lives in subsystem-owned pools/arenas with explicit lifecycle; ECS components store only small POD handles/indices/IDs into that state. 2.5) Scalability Rules (100K+): Use dirty flags to skip unchanged data. Keep rare state sparse (don’t spread “rare” data across every entity/component). Update spatial indices incrementally (only moved/changed entities). Never iterate “all entities” unconditionally: iterate only relevant subsets (active/visible/AI-owned), and make subset selection explicit in system design. 2.75) Spatial Contract (flagship-critical): core::spatial is first-class scaling primitive (grid/quadtree, chunked maps, spatial queries). For systems depending on proximity/visibility/perception/broad-phase: brute-force scans are forbidden; systems must query core::spatial and/or derived registries (RenderQueue, ActiveAISet), updated incrementally and driven by dirty flags. 3) Sprite / Rendering Design: SpriteComponent is data-only: TextureKey (RuntimeKey32) textureKey; sf::IntRect textureRect; float zOrder; 0-sized textureRect means “full texture”. No sf::Sprite in components. RenderSystem iterates (TransformComponent, SpriteComponent); resolves textures via ResourceManager; uses transient sprite(s); draw order by zOrder; extend later with culling/batching/spatial. 4) Deterministic Ordering: Use external std::vector<entt::entity> + std::sort. Pattern: build view once; collect entities into scratch vector; comparator uses view.get<T>(entity); reuse scratch vectors each frame. registry.sort<T>() only for small mostly static sets and only when data is dirty; never every frame on large sets; never sort same component by different criteria in different systems. Canonical ordering: when ordering matters for determinism, use explicit stable ordering key (e.g., stable ID) and prefer shared helpers over ad-hoc per-system sorting. 5) World Usage Best Practices: Prefer in-place addComponent<T>(e, args...) and explicitly init non-empty components. Prefer tryGetComponent<T>() for check+use (single lookup); use getComponent<T>() for fail-fast (assert). Use view()/view() const for iteration; group() only when needed for hot paths. Do not rely on implicit iteration order when order matters (collect+sort). Use registry() only as escape hatch for advanced EnTT features.
SFML1 / SkyGuard — Canonical Context (Part 10): Turn-Based Simulation Cadence (layered, configurable): Turn-based simulation uses layered cadences: tactical systems run every turn; operational/strategic systems run less frequently (configurable) to keep worst-case turn times bounded and predictable.
SFML1 / SkyGuard — Canonical Context (Part 11): AI Architecture (data-driven + budgeted): AI is fully data-driven (doctrines/focus/goals in JSON). Decision hierarchy: strategic — utility scoring chooses active goals/foci; planning — GOAP/HTN-like expansion into concrete actions; tactical — behavior trees with lightweight utility scoring. AI execution model: enforced by an AI scheduler with explicit budgets (per empire/front/group); time-sliced execution with explicit resume points (continue next tick/turn when budget is hit); mandatory metrics (processed/skipped, time spent) exposed to debug overlay/profiler. Performance is enforced via caching (utility scores, influence maps, pathfinding) and hierarchical execution (strategic sets goals, tactical executes without re-evaluating strategy every turn).
SFML1 / SkyGuard — Canonical Context (Part 12): Determinism, Saves, Replay: Simulation determinism is a first-class requirement (fixed system update order, stable iteration rules, seeded RNG, avoid nondeterministic operations). Save format is versioned with migrations. Large-scale saves: delta saves (only changed components), compressed/sparse serialization, chunked loading/streaming (entity→chunk mapping + manifest; per-chunk compression). Save metadata includes entity/component counts, playtime, and a game-state hash. Resource registry reproducibility: save header stores the exact source/mod list and deterministic load order (or an equivalent hash) used to build the resource registry; on load, rebuild the registry in the save order, or warn and refuse to load if active sources/mods/order are incompatible.
SFML1 / SkyGuard — Canonical Context (Part 13): Performance Targets & Budgets (non-negotiable framing): Fundamental rule: performance is enforced via explicit budgets and LOD/time-slicing, not by hoping “fast enough”. Hard constraints: no recurring “hypothetical-case” checks in hot paths (Release); diagnostics exist but must be cheap-by-default (Debug/Trace gated, rate-limited, or moved to overlay/profiler).
SFML1 / SkyGuard — Canonical Context (Part 14): Job-Friendly Design & Concurrency: Implementation is single-threaded for now, but job-friendly by design: no global mutable state; data owned by World/Game; systems grouped into phases/layers for future scheduling. Phase-based execution: fixed phase order (deterministic); system dependencies derived from declared component access (read vs write); may run single-threaded today, but API and dataflow must remain job-friendly. Hot-Reload: Planned file watching + hot-reload for JSON configs (engine/gameplay/AI) with validation, without restarting the game.
SFML1 / SkyGuard — Canonical Context (Part 15): Main idea ALWAYS: “Performance-first!” If there is any doubt, treat performance as the primary priority and act accordingly.
SFML1 / SkyGuard — Canonical Context (Part 16): JSON Options by Data Provenance. Choose JSON parse/validation options based on where the data comes from. Hand-authored configs (engine/gameplay): reject duplicate keys for determinism/authoring safety; do structural checks once at the boundary; prefer field-level *WithIssue parsers as single source of truth and keep SchemaPolicy::Skip to avoid duplicate checks; enable SchemaPolicy::Validate only for all-or-nothing configs or when *WithIssue coverage is missing. SFML1 / SkyGuard — Canonical Context (Part 19): Resource ID Stability (Mods/Saves): The stability contract for resources is defined by Canonical Key Strings: canonical keys must not be removed or renamed for mod compatibility. StableKey64 is derived deterministically from canonical keys (xxHash3_64, fixed seed 0) and provides cross-version stability for saves, replays, and mod manifests; changing derivation is a breaking change that requires an explicit migration strategy. Compile-time enums are not a mod/save contract; they may exist only as local convenience in prototypes. Legacy (pre-Resource Registry v1): enum IDs were never removed or reordered; this rule is deprecated.
SFML1 / SkyGuard — Canonical Context (Part 20): Component Size Documentation — component headers must document size in bytes in Purpose field for cache-line awareness. Format: "Purpose: [description] (hot path, N bytes)". Example: "Purpose: ID-based sprite component (hot path, 40 bytes)". Target <64 bytes for hot components (1 cache line). Helps detect bloat during refactoring.
SFML1 / SkyGuard — Canonical Context (Part 21): Move Semantics Manual Zeroing — custom types with primitive counters must manually zero source in move operations. Default move copies primitives but leaves source with stale values, causing invariant violations (e.g., mAliveEntityCount desync). Pattern: World(World&& other) noexcept : mRegistry(std::move(other.mRegistry)), mAliveEntityCount(other.mAliveEntityCount) { other.mAliveEntityCount = 0; /* CRITICAL */ }. See world.h:115-119 for example. This prevents moved-from object destructors from triggering asserts.
SFML1 / SkyGuard — Canonical Context (Part 22): EnTT Signals for Cleanup — use EnTT on_destroy signals for automatic resource cleanup to avoid manual bookkeeping. Pattern: registry.on_destroy<SpatialHandleComponent>().connect<&SpatialIndexSystem::onDestroyed>(*this). This is RAII for ECS—cleanup happens automatically without manual calls in destroy paths. Never manually unregister in destroyEntity(); let signals handle it. See spatial_index_system.cpp:96-98 for canonical example.
SFML1 / SkyGuard — Canonical Context (Part 23): Deterministic Resource Registry Build & Loading — resource registry build and resource loading must be deterministic for reproducible behavior. Deterministic order is defined by StableKey64 total order; RuntimeKey32 indices are assigned in that deterministic order. Duplicate canonical keys within one source file (same canonical key appears twice in the same JSON source) → PANIC (authoring error). Duplicate canonical keys across sources → expected and resolved deterministically by override policy (layers + loadOrder). Benefits: stable logs (easier diff between runs), reproducible bugs (same allocation pattern), predictable allocation/loading behavior, and deterministic saves/replays. Legacy (pre-Resource Registry v1): deterministic loading sorted by enum ID (preloadAllSorted) is deprecated.
SFML1 / SkyGuard — Canonical Context (Part 24): Entity ID Policy (Runtime vs Persistent) — entt::entity is RUNTIME ONLY (generation counter, may be reused after destroy). NEVER serialize entt::entity to disk or network. Saves/network require STABLE IDs (separate uint64 + bidirectional mapping table entt::entity ↔ StableID). Use toUint(entity) for logs/debug only, never for persistence. See entity.h:66-68 for detailed rationale. Implementation: when 4X save system is built (RoadMap).
